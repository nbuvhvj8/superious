# Tool Infrastructure: Building the 80% Brain
## Research Paper · Superious AI Agent Tool Layer

> "Only 1.6% of Claude Code's codebase is AI decision logic. The other 98.4% is
> deterministic infrastructure — permission gates, context management, tool routing,
> and recovery logic."
> — VILA-Lab, *Dive into Claude Code* (2026)

---

## Executive Summary

This document is the architectural blueprint for Superious's tool layer — the
infrastructure that will constitute 80–85% of the agent's decision-making body,
with the AI model and user approval making up the remaining 15–20%. The design
is directly informed by the leaked Claude Code 512K-line TypeScript codebase
(March 2026), the VILA-Lab systematic analysis, and cross-analysis from George
Sung's LLM traffic traces and WaveSpeed's deep dive.

The core thesis: **the model is the least important part of an intelligent agent.**
What separates a toy chatbot from a production-grade research engine is the
harness — the deterministic, composable, permission-gated, memory-aware tool
system surrounding it. Build the harness well, and any capable model becomes
powerful. Build it poorly, and even the best model fails.

## Part I: The Claude Code Infrastructure Blueprint

### 1.1 The Real Architecture Ratio

The leaked Claude Code source revealed a counterintuitive truth about how
Anthropic actually built their flagship agent. The `query.ts` agentic loop —
the part that actually talks to Claude — is a 1,729-line `while(true)` loop.
The surrounding infrastructure — permissions, tool routing, context compaction,
memory, recovery logic — spans 510,000+ lines. That is the 98.4/1.6 ratio.

The architecture decomposes into 7 components across 5 layers:

```
User Input
    ↓
Interfaces (REPL / SDK / IDE)
    ↓
Agent Loop (query.ts — the while(true))
    ↓
Permission System (deny-first; strictest rule wins)
    ↓
Tools (~40 built-in + dynamic MCP tools)
    ↓
State & Persistence (memory.md, CLAUDE.md, session transcripts)
    ↓
Execution Environment (bash, filesystem, network)
```

The most critical insight: **the Agent Loop is the simplest layer.** It is just
a loop that yields streaming events. Every hard problem — context overflow,
tool discovery, permission arbitration, memory consolidation — is solved
*outside* the model, in deterministic code.

### 1.2 The Five Compaction Layers (Context Management)

Claude Code's context compaction runs 5 strategies *before every model call*:

1. **Message deduplication** — identical or near-identical tool results are
   collapsed into a single summary message
2. **Tool result truncation** — tool outputs over a threshold are trimmed with
   a summary prepended
3. **Tombstoning** — cancelled or errored tool calls are replaced with
   lightweight marker messages
4. **Summarization sweep** — when context > 70% full, older turns are replaced
   with a summary generated by a smaller model (Haiku in Claude Code's case)
5. **Forced compaction** — at 90% context fill, an aggressive sweep prunes to
   the essential working set

The implication for Superious: **context management is not optional.** Every
research job will exhaust context if web-search results, source summaries, and
script drafts are naïvely concatenated. The tool layer must manage what the
model sees, not just pass everything through.

### 1.3 The Tool Discovery Pattern (ToolSearchTool)

When Claude Code's tools grew beyond ~20 built-in + dozens of MCP tools, their
definitions alone consumed thousands of tokens per request. The solution was
elegant:

```typescript
// tools.ts:247-249
// Include ToolSearchTool when tool search might be enabled
...(isToolSearchEnabledOptimistic() ? [ToolSearchTool] : []),
```

Instead of loading all tools upfront, **only a single meta-tool is provided** —
a "tool that finds tools." When the model needs to search files, it first calls
`ToolSearchTool("file search")`, which returns the definition of `ReadTool` or
`GrepTool`. This deferred loading pattern has two benefits:

- **Token efficiency**: only loaded tool definitions appear in the context
- **Composability**: tools can be added without expanding the base system prompt

For Superious, this pattern becomes the **Tool Registry** — a central store of
all 20 tools with lazy loading, invoked by a `tool_search` root tool.

### 1.4 The Permission System (Deny-First)

Claude Code uses a deny-first permission model: every operation is denied by
default and must be explicitly allowed by a rule. Rules have priorities, and the
strictest rule always wins. The system has two tiers: Standard (read + limited
write) and Elevated (full execution, remote operations).

The security architecture has 7 independent layers:
1. Input validation before any tool invocation
2. Path sandboxing (no reads outside project root by default)
3. Allowlist/blocklist for bash commands
4. Per-operation confirmation prompts (with approval fatigue mitigation)
5. Tool result sanitisation before re-entry into context
6. Rate limiting on expensive operations
7. Audit logging of every tool call

### 1.5 The Three-Layer Memory Architecture

The leaked code confirmed a three-layer memory system:

| Layer | Mechanism | Scope |
|-------|-----------|-------|
| **Static Config** | `CLAUDE.md` in project root | Persistent across all sessions |
| **Working Memory** | `memory.md` with pointer index | Persists across sessions, consolidated periodically |
| **Active Retrieval** | Grep-based search + `MEMORY.md` pointers | On-demand, per-turn |

The `autoDream` background service consolidates working memory:
- Triple gate: 24 hours elapsed + 5+ sessions accumulated + file advisory lock acquired
- Cheapest checks run first to minimise overhead
- A forked sub-agent performs the consolidation, not the main loop

This is the model for Superious's **job memory** — each job should persist a
`job-state.json` file that survives worker restarts and acts as the equivalent
of `memory.md`.

### 1.6 The Async Generator Agent Loop Pattern

Claude Code's agentic loop uses `async function*` (async generators) for all
streaming events. This is superior to EventEmitter or callback patterns because:

- **Single channel**: stream events, termination, and errors all flow through
  one `for await...of` loop
- **Backpressure**: the consumer controls the rate of consumption
- **Composability**: generators can be chained and filtered

For LangGraph.js in Superious, this maps directly to LangGraph's streaming
output mode. Every node should yield events, not return them, and the API layer
should forward those events as SSE to the browser.

## Part II: The 20 Superious Built-in Tools

### Design Principles

These tools follow the Claude Code harness model:

1. **Deterministic over probabilistic** — tools do real work; the AI decides
   *which* tools and *in what order*, not *how* each tool works internally
2. **Single responsibility** — one tool, one capability, no side effects outside
   its declared scope
3. **Structured I/O** — every tool has a typed input and typed output; no raw
   strings at the boundary
4. **Permission-gated** — every destructive or expensive operation requires an
   explicit capability grant
5. **Observable** — every tool call is logged to the audit trail with duration,
   inputs (sanitised), and result summary
6. **Compatible stack** — all tools are TypeScript-native and designed for
   LangGraph.js nodes in a Hono.js API on Railway, with Supabase as the
   persistence layer

## Part III: Tool Execution Architecture

### 3.1 The Tool Harness (Wraps All 20 Tools)

Every tool call flows through this wrapper before execution:

```typescript
async function executeTool<I, O>(
  toolName: string,
  input: I,
  context: ToolUseContext
): Promise<ToolResult<O>> {
  // 1. Permission Gate (deny-first check)
  const perm = await permissionGate.check(context.userId, toolName, input);
  if (!perm.granted) return { success: false, error: 'PERMISSION_DENIED' };

  // 2. Input validation (Zod schema)
  const parsed = toolSchemas[toolName].safeParse(input);
  if (!parsed.success) return { success: false, error: 'INVALID_INPUT' };

  // 3. Rate limit check
  const quota = await rateLimiter.check(context.userId, toolName);
  if (!quota.allowed) return { success: false, error: 'RATE_LIMITED' };

  // 4. Execute
  const startMs = Date.now();
  try {
    const output = await tools[toolName].execute(parsed.data, context);
    const durationMs = Date.now() - startMs;

    // 5. Audit log (fire-and-forget)
    auditLogger.log({ jobId: context.jobId, event: {
      type: 'tool_call', tool: toolName, durationMs, success: true,
      inputSummary: sanitise(input), outputSummary: summarise(output)
    }});

    return { success: true, data: output };
  } catch (err) {
    // 6. Error classification + recovery
    const recovery = errorClassifier.classify(err, { tool: toolName, jobId: context.jobId });
    return { success: false, error: recovery.category, recovery };
  }
}
```

### 3.2 LangGraph Node Mapping

Each LangGraph node corresponds to one or more tools:

```
Graph Node            → Tools Invoked
─────────────────────────────────────────────────────────────
query_planner         → topic_classifier, query_planner
web_researcher        → web_search (x N queries), source_fetch
source_ranker         → source_ranker, source_deduplicator
screenshot_dispatcher → screenshot_capture (parallel, BullMQ)
source_summariser     → source_summariser (per source)
context_compactor     → context_manager (before script_writer call)
script_writer         → [Claude Sonnet model call]
script_formatter      → script_formatter, b_roll_suggester
state_saver           → job_state_manager, sse_streamer
```

The model is invoked **once** — in `script_writer`. Every other node is
deterministic tool execution. This is the harness model: 8 deterministic nodes,
1 model node.

### 3.3 The Tool-to-Token Budget

Knowing approximately how many tokens each tool returns helps the
`context_manager` decide what to compact first:

| Tool | Estimated output tokens |
|------|------------------------|
| `web_search` (10 results) | ~800 |
| `source_fetch` (one page) | ~4,000 |
| `source_summariser` | ~300 |
| `query_planner` | ~150 |
| `source_ranker` | ~200 |
| `script_formatter` (full script) | ~3,000–6,000 |

Context budget before script_writer call: allocate 40K tokens for the script
writer's working context. Source summaries should total no more than 8,000 tokens
(~26 sources × 300 tokens, trimmed to 8 sources × 300 = 2,400 tokens in practice).
